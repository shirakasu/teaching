# ソフトウェア設計 - オブジェクト指向プログラミング

## オブジェクト指向プログラミングとは

英語はObject-Oriented Programing．
名前の通り，システムを構成する要素をオブジェクトとして捉え，オブジェクト同士の
関係性を作ることでより現実世界に即したシステムを設計する考え方．
オブジェクトとは**データ構造と処理（メソッド）をセットにしたモノ**と考えてよい．

オブジェクト指向プログラミングができる前は，**手続き型プログラミング**が用いられる．
これは処理の手順をもとにシステムを構成していくプログラミング方法．
つまりやることを順番に並べて記述する方法．

## オブジェクト指向 vs 手続き型

### 手続き型

例えば，「Aのデータを取ってくる」→「Bの計算をする」→「Cの画面に出力する」というもの．
すなわちデータと処理がバラバラに存在する．

デメリットとして，プログラムが大規模になると，どこでどのデータが書き換えられたか追跡しにくくなる．
このため一部を修正すると全く関係ない場所でバグが出ることが増える．
（ちなみにこの現象をスパゲッティコードと呼ぶ）

## 抽象と具体を活用した継承

Point: 
is-aの原則が成立しない場合は，継承は使うべきではない．
つまり() is a ().の文章が成立しない場合はダメ．

良い例：Car - Vehicle. Cat - Pet.
悪い例：Engine is a Car. Cat - tail.

## オブジェクト指向におけるアンチパターン

**巨大クラス**

1つのクラスにすべての機能を詰め込んだ設計であり，管理がかえって複雑になる．
クラス分けを行う際には，責務の分割を意識して行おう．

**継承の乱用**

やみくもくクラスを継承しすぎてしまうと，変更が難しくなる．
オブジェクトの組み合わせを活用しよう．

**過度な抽象化**

抽象クラスやインターフェースを増やしすぎると，コードが複雑になってしまう．
必要な部分だけを抽象化すると良い．
